# 任务计划：修复多维表格生成应用 WrongRequestBody

## 目标

定位并修复“根据飞书多维表格生成系统应用”时报 `WrongRequestBody` 的问题。

## 约束

- 中文回复。
- 不运行 `npm run build`。
- 不运行任何 test 测试命令。
- 先定位根因，再做最小改动。
- 优先复用现有飞书客户端和动态应用生成链路。

## 阶段

1. [完成] 梳理生成应用 API、飞书客户端、动态应用构建链路。
2. [完成] 对照飞书多维表格记录读取接口，定位错误请求体来源。
3. [完成] 增加错误定位信息并兼容可选 query 导致的 WrongRequestBody。
4. [完成] 修复列数据表接口的 WrongRequestBody。
5. [完成] 做非 build/test 静态复核。

## 当前判断

- 报错 `WrongRequestBody` 发生在生成应用时读取样例记录阶段，链路为：
  `/api/ai-apps/generate` -> `buildDynamicAppFromFeishu()` -> `searchFeishuBitableRecords()`。
- 现有代码用 `POST /records/search`，并构造 JSON body；生成应用只是读取记录，不需要搜索接口的请求体能力。
- 飞书“列出记录”接口是 `GET /bitable/v1/apps/{app_token}/tables/{table_id}/records`，`field_names/page_size/page_token/view_id` 作为 query 参数传入，更适合本场景。
- 用户反馈改 GET 后仍为 `WrongRequestBody`，说明错误可能来自：
  1. 运行服务尚未加载新代码；
  2. `field_names/view_id` 这类可选 query 在该表/权限下不被接受；
  3. 其它飞书接口返回同名错误但前端未显示接口位置。
- 第二轮修复方向：保留 GET `/records`，但对 `WrongRequestBody` 自动降级为最小 query；同时错误信息带上脱敏接口路径，方便继续定位。
- 最新错误已明确来自 `GET /bitable/v1/apps/{app_token}/tables`。优先去掉 `page_size` 这类非必要 query，用最小请求列数据表；字段列表也同步改为最小 query，避免同类问题延后出现。

# 任务计划：考试页首次进入卡顿排查

## 目标

定位并优化“点击开始/继续考试后首次进入考试页面很慢”的原因，判断是否属于 Next.js 本身问题，并在不运行 build/test 的前提下做最小清晰改动。

## 约束

- 不运行 `npm run build`。
- 不运行任何 test 测试命令。
- 只在 `gjj` 项目内开发。
- 先定位根因，再修改。

## 阶段

1. [完成] 排查学生考试列表、考试详情页、store 持久化调用链。
2. [完成] 判断首进卡顿的主要阻塞点。
3. [完成] 实现最小优化：避免点击后等待持久化完成，提前预取考试页。
4. [完成] 静态自查改动和未运行验证项。

## 发现/假设

- `app/student/page.tsx` 的新考试入口在 `await startAttempt(newAttempt)` 后才 `router.push(...)`。
- `startAttempt` 会先同步更新 Zustand，再异步 `PATCH /api/exam-attempts/:id` 保存 SQLite。
- 因此新考试首次点击会被 API route/SQLite 首次初始化/开发环境懒编译阻塞。
- 继续考试分支已经直接 `router.push`，仍慢时更像 Next.js dev 的动态路由首次编译/加载。
- 这不是“Next.js 不能用”的问题，更像开发模式首进动态路由 + 点击链路等待后端保存叠加。
- 用户进一步反馈“所有页面首次打开都慢”，这符合 Next.js `dev` 模式按路由懒编译的特征：每个页面第一次请求慢，之后缓存编译结果就快。
- 已修改 `app/student/page.tsx`：
  - 新建考试不再等待 `startAttempt` 的 SQLite 保存完成，先写入前端 store 并立即跳转。
  - 保存失败改为后台 toast 提示，不阻塞进入考试。
  - 对已有进行中考试执行 `router.prefetch('/student/exam/:attemptId')`。
  - 点击后按钮显示“进入中...”并禁用，减少重复点击。
- 已修改 `components/ClientLayout.tsx`：
  - 登录并加载数据后，对主要静态页面执行 `router.prefetch`。
  - 在开发模式额外用后台 `fetch` 预热主要页面和动态路由占位路径，让 Next dev 提前编译页面。
